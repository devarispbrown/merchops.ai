## Prerequisites

Before deploying MerchOps to any platform, ensure you have:

### Required Accounts

1. **PaaS Platform Account** (Render/Railway/Fly.io/Vercel)
2. **Shopify Partner Account** for OAuth credentials
3. **AI Provider Account** (Anthropic/OpenAI) for LLM capabilities
4. **Email Provider Account** (Resend/SendGrid) for transactional emails
5. **Sentry Account** (optional but recommended) for error tracking

### Required Tools

```bash
# Node.js and pnpm
node -v  # v20.x or higher
pnpm -v  # v8.15.0 or higher

# Git
git --version

# Platform CLI (optional but helpful)
# Render: No CLI needed
# Railway: npm install -g @railway/cli
# Fly.io: curl -L https://fly.io/install.sh | sh
# Vercel: npm install -g vercel
```

### Prepare Your Codebase

```bash
# Clone and install dependencies
git clone <your-repo-url>
cd merchops.ai
pnpm install

# Verify tests pass
pnpm lint
pnpm typecheck
pnpm test

# Ensure migrations are ready
pnpm prisma:validate
```

## Environment Variables

See **[Environment Variables Reference](./env-vars.md)** for complete documentation of all required and optional environment variables.

**Critical Variables:**
- `DATABASE_URL` - PostgreSQL connection string
- `REDIS_URL` - Redis connection string
- `NEXTAUTH_SECRET` - Session encryption key
- `NEXTAUTH_URL` - Your application URL
- `SHOPIFY_CLIENT_ID` / `SHOPIFY_CLIENT_SECRET` - OAuth credentials
- `ENCRYPTION_KEY` - For encrypting Shopify tokens
- `ANTHROPIC_API_KEY` or `OPENAI_API_KEY` - AI provider credentials

## Platform-Specific Guides

### Render (Recommended)

Render provides the simplest deployment experience with built-in PostgreSQL, Redis, and zero-downtime deploys.

#### Why Render?

- Blueprint-based infrastructure as code
- Built-in PostgreSQL and Redis (no external setup)
- Automatic SSL certificates
- Zero-downtime deployments
- Simple scaling
- Good free tier for testing

#### Deployment Steps

1. **Fork/Push Repository to GitHub**
   ```bash
   git remote add origin <your-github-repo>
   git push -u origin main
   ```

2. **Create Render Account**
   - Go to [render.com](https://render.com)
   - Sign up with GitHub

3. **Deploy Using Blueprint**
   - Click "New" → "Blueprint"
   - Connect your GitHub repository
   - Render will detect `render.yaml` automatically
   - Review the services:
     - `merchops-web` (Next.js app)
     - `merchops-worker` (Background jobs)
     - `merchops-db` (PostgreSQL)
     - `merchops-redis` (Redis)

4. **Configure Environment Variables**

   In Render Dashboard, set the following variables (marked `sync: false` in blueprint):

   ```bash
   # Application URL (use your Render URL)
   NEXTAUTH_URL=https://merchops-web.onrender.com

   # Shopify OAuth
   SHOPIFY_CLIENT_ID=your_shopify_client_id
   SHOPIFY_CLIENT_SECRET=your_shopify_client_secret

   # AI Provider
   ANTHROPIC_API_KEY=your_anthropic_key
   # OR
   OPENAI_API_KEY=your_openai_key

   # Email Provider
   EMAIL_PROVIDER_API_KEY=your_resend_or_sendgrid_key

   # Optional: Sentry
   SENTRY_DSN=your_sentry_dsn
   ```

   **Note:** `NEXTAUTH_SECRET`, `ENCRYPTION_KEY`, and `SHOPIFY_WEBHOOK_SECRET` are auto-generated by Render.

5. **Run Database Migrations**

   After first deployment, run migrations via Render Shell:

   - Go to `merchops-web` service
   - Click "Shell" tab
   - Run:
     ```bash
     pnpm prisma migrate deploy
     ```

6. **Verify Deployment**
   - Visit your app URL: `https://merchops-web.onrender.com`
   - Check health endpoint: `https://merchops-web.onrender.com/api/health`
   - Sign up and test Shopify OAuth flow

#### Render Configuration Tips

**Database Connection Pooling:**
```
# Render PostgreSQL supports native pooling
# No additional configuration needed
```

**Scaling:**
```bash
# Upgrade plans in Render Dashboard:
# - Web: Standard plan for 2GB RAM, autoscaling
# - Worker: Scale to multiple instances
# - DB: Standard plan for 4GB RAM, daily backups
# - Redis: Standard plan for 1GB memory
```

**Custom Domain:**
```bash
# In Render Dashboard:
# - Go to Settings → Custom Domains
# - Add your domain
# - Update DNS records
# - Update NEXTAUTH_URL to your domain
```

### Railway.app

Railway offers excellent developer experience with built-in observability and simple pricing.

#### Deployment Steps

1. **Install Railway CLI**
   ```bash
   npm install -g @railway/cli
   railway login
   ```

2. **Create New Project**
   ```bash
   railway init
   # Link to existing project or create new
   ```

3. **Add PostgreSQL**
   ```bash
   railway add postgresql
   ```

4. **Add Redis**
   ```bash
   railway add redis
   ```

5. **Deploy Application**
   ```bash
   railway up
   ```

6. **Configure Environment Variables**
   ```bash
   # Set via Railway Dashboard or CLI
   railway variables set NEXTAUTH_SECRET=$(openssl rand -base64 32)
   railway variables set NEXTAUTH_URL=https://your-app.railway.app
   railway variables set SHOPIFY_CLIENT_ID=your_id
   railway variables set SHOPIFY_CLIENT_SECRET=your_secret
   railway variables set ENCRYPTION_KEY=$(openssl rand -hex 32)
   railway variables set ANTHROPIC_API_KEY=your_key
   railway variables set EMAIL_PROVIDER_API_KEY=your_key
   ```

7. **Add Worker Service**

   Railway doesn't auto-detect workers from render.yaml. Add manually:

   - In Railway Dashboard → "New Service"
   - Choose "From Existing Repo"
   - Set custom start command: `pnpm workers`
   - Share environment variables with web service

8. **Run Migrations**
   ```bash
   railway run pnpm prisma migrate deploy
   ```

#### Railway Tips

**Automatic Deploys:**
```bash
# Enable GitHub integration
railway link
railway up --detach
```

**Monitoring:**
```bash
# Railway includes built-in metrics
# View in Dashboard → Metrics tab
```

**Estimated Cost:**
- Hobby plan: $5/month + usage
- Free tier: $5 credit/month
- Web + Worker + DB + Redis ≈ $15-30/month

### Fly.io

Fly.io offers global edge deployment with excellent performance and control.

#### Deployment Steps

1. **Install Fly CLI**
   ```bash
   curl -L https://fly.io/install.sh | sh
   flyctl auth login
   ```

2. **Create App Configuration**
   ```bash
   flyctl launch
   # Follow prompts, select:
   # - Region closest to users
   # - PostgreSQL database
   # - Redis (via Upstash integration)
   ```

3. **Configure fly.toml**

   Fly generates `fly.toml`. Update it:

   ```toml
   app = "merchops"
   primary_region = "sjc"

   [build]
     [build.args]
       NODE_VERSION = "20"

   [env]
     NODE_ENV = "production"
     PORT = "8080"

   [http_service]
     internal_port = 3000
     force_https = true
     auto_stop_machines = false
     auto_start_machines = true
     min_machines_running = 1

   [[services]]
     protocol = "tcp"
     internal_port = 3000

     [[services.ports]]
       port = 80
       handlers = ["http"]
       force_https = true

     [[services.ports]]
       port = 443
       handlers = ["tls", "http"]

     [services.concurrency]
       type = "connections"
       hard_limit = 1000
       soft_limit = 500

   [[vm]]
     memory = '1gb'
     cpu_kind = 'shared'
     cpus = 1
   ```

4. **Set Secrets**
   ```bash
   flyctl secrets set NEXTAUTH_SECRET=$(openssl rand -base64 32)
   flyctl secrets set NEXTAUTH_URL=https://merchops.fly.dev
   flyctl secrets set SHOPIFY_CLIENT_ID=your_id
   flyctl secrets set SHOPIFY_CLIENT_SECRET=your_secret
   flyctl secrets set ENCRYPTION_KEY=$(openssl rand -hex 32)
   flyctl secrets set ANTHROPIC_API_KEY=your_key
   flyctl secrets set EMAIL_PROVIDER_API_KEY=your_key
   flyctl secrets set SHOPIFY_WEBHOOK_SECRET=$(openssl rand -base64 32)
   ```

5. **Add Database and Redis URLs**
   ```bash
   # Fly provides these automatically
   # Verify with:
   flyctl secrets list
   ```

6. **Deploy Background Worker**

   Create separate worker app:

   ```bash
   flyctl apps create merchops-worker
   ```

   Create `fly-worker.toml`:
   ```toml
   app = "merchops-worker"
   primary_region = "sjc"

   [build]
     [build.args]
       NODE_VERSION = "20"

   [env]
     NODE_ENV = "production"

   [[services]]
     # No HTTP service for worker
   ```

   Deploy worker:
   ```bash
   flyctl deploy --config fly-worker.toml --command "pnpm workers"
   ```

7. **Run Migrations**
   ```bash
   flyctl ssh console
   pnpm prisma migrate deploy
   exit
   ```

8. **Deploy**
   ```bash
   flyctl deploy
   ```

#### Fly.io Tips

**Multi-Region Deployment:**
```bash
# Scale to multiple regions
flyctl scale count 2 --region sjc,ord
```

**Monitoring:**
```bash
# Built-in Prometheus metrics
flyctl dashboard
```

**Estimated Cost:**
- Hobby plan: $0 for 3 small VMs (256MB)
- Production: $10-20/month for web + worker
- Add $5-15/month for DB and Redis

### Vercel + External Services

Deploy the Next.js app on Vercel while using external managed services for database and Redis.

#### Why This Approach?

- Best Next.js hosting (made by Vercel)
- Excellent DX and preview deployments
- Need external DB/Redis (Vercel doesn't provide these)
- Good for teams already on Vercel

#### Deployment Steps

1. **Set Up External Services**

   **Option A: Railway for DB + Redis**
   ```bash
   railway init
   railway add postgresql
   railway add redis
   # Get connection strings from Railway Dashboard
   ```

   **Option B: Supabase (PostgreSQL) + Upstash (Redis)**
   - Supabase: [database.new](https://database.new)
   - Upstash: [console.upstash.com](https://console.upstash.com)

2. **Deploy to Vercel**
   ```bash
   # Install Vercel CLI
   npm install -g vercel

   # Deploy
   vercel

   # Follow prompts:
   # - Link to existing project or create new
   # - Set build command: pnpm build
   # - Set output directory: .next
   ```

3. **Configure Environment Variables**

   In Vercel Dashboard → Settings → Environment Variables:

   ```bash
   DATABASE_URL=postgresql://user:pass@host:5432/db
   REDIS_URL=redis://user:pass@host:6379
   NEXTAUTH_SECRET=<generate>
   NEXTAUTH_URL=https://your-app.vercel.app
   SHOPIFY_CLIENT_ID=<your-id>
   SHOPIFY_CLIENT_SECRET=<your-secret>
   ENCRYPTION_KEY=<generate>
   ANTHROPIC_API_KEY=<your-key>
   EMAIL_PROVIDER_API_KEY=<your-key>
   SHOPIFY_WEBHOOK_SECRET=<generate>
   LOG_LEVEL=info
   NODE_ENV=production
   ```

4. **Deploy Background Worker Separately**

   **Problem:** Vercel doesn't support long-running background workers.

   **Solution:** Deploy worker to Railway/Render/Fly.io

   Example with Railway:
   ```bash
   # Create new Railway service
   railway add
   # Select "Empty Service"
   # Set start command: pnpm workers
   # Share environment variables from Vercel
   ```

5. **Run Migrations**
   ```bash
   # Via Vercel CLI
   vercel env pull .env.production
   pnpm prisma migrate deploy
   ```

6. **Redeploy**
   ```bash
   vercel --prod
   ```

#### Vercel Tips

**Serverless Function Timeout:**
```bash
# Vercel has 10s timeout on Hobby, 60s on Pro
# Ensure webhook handlers complete quickly
# Move heavy processing to background jobs
```

**Preview Deployments:**
```bash
# Every PR gets a preview URL
# Use separate database for previews
vercel env add DATABASE_URL preview
```

**Estimated Cost:**
- Vercel Hobby: Free for personal projects
- Vercel Pro: $20/month
- External DB: $5-15/month (Railway/Supabase)
- External Redis: $0-10/month (Upstash)
- Worker hosting: $5-10/month (Railway/Render)
- **Total:** $30-55/month

## Post-Deployment Checklist

After deploying to any platform, complete this checklist:

### 1. Health Checks

```bash
# Application health
curl https://your-app.com/api/health

# Expected response:
# {"status":"ok","timestamp":"2024-01-23T..."}
```

### 2. Database Verification

```bash
# Via platform shell or local connection
pnpm prisma studio

# Verify tables exist:
# - Workspace
# - User
# - ShopifyConnection
# - Event
# - Opportunity
# - ActionDraft
# - Execution
```

### 3. Redis Verification

```bash
# Check Redis connection
# Via platform shell:
redis-cli -u $REDIS_URL ping
# Should return: PONG
```

### 4. Background Worker Verification

Check worker logs for:
```
Worker started successfully
Connected to Redis
Listening for jobs on queues: shopify-sync, event-computation, opportunity-generation, action-execution, outcome-resolution
```

### 5. Shopify OAuth Flow

1. Sign up for new account
2. Click "Connect Shopify"
3. Complete OAuth flow
4. Verify `shopify_connections` table has entry
5. Check webhook registration:
   ```bash
   # In Shopify Admin → Settings → Notifications → Webhooks
   # Should see webhooks for:
   # - orders/create
   # - orders/paid
   # - products/update
   # - inventory_levels/update
   ```

### 6. Webhook Delivery Test

1. In Shopify Admin, create a test order
2. Check logs for webhook receipt
3. Verify `events` table has new entry

### 7. Opportunity Generation Test

1. Wait for initial sync to complete (check logs)
2. Trigger opportunity conditions:
   - Low inventory product
   - Inactive customer (if test data available)
3. Check `opportunities` table
4. Verify opportunities appear in dashboard

### 8. Action Execution Test

1. Find an opportunity in dashboard
2. Review draft action
3. Approve action
4. Verify execution completes
5. Check `executions` table for record

### 9. Monitoring Setup

Configure alerts for:
- Application errors (Sentry)
- Database connection failures
- Redis connection failures
- Webhook delivery failures
- Job failures

### 10. Performance Baseline

Establish baseline metrics:
- Dashboard load time: < 2s
- Webhook processing: < 5s p95
- Opportunity generation: < 5 min p95
- Action execution: < 30s p95

## Monitoring and Observability

### Built-in Observability

MerchOps includes structured logging with correlation IDs:

```typescript
// All requests tagged with correlation ID
// Example log output:
{
  "level": "info",
  "msg": "Opportunity generated",
  "correlationId": "req_123abc",
  "opportunityId": "opp_456def",
  "priority": "high"
}
```

### Sentry Integration

Configure Sentry for error tracking:

```bash
# Set in environment
SENTRY_DSN=https://your-sentry-dsn@sentry.io/project
```

Sentry captures:
- Application errors
- Unhandled promise rejections
- Performance transactions
- User context (workspace ID, anonymized)

### Platform-Specific Monitoring

**Render:**
- Built-in metrics dashboard
- Log aggregation
- Auto-scaling alerts

**Railway:**
- CPU/Memory metrics
- Log search
- Deploy notifications

**Fly.io:**
- Prometheus metrics
- Grafana dashboards
- Global health checks

**Vercel:**
- Analytics dashboard
- Function logs
- Web Vitals

### Custom Dashboards

Query logs via platform log viewers:

```bash
# Find slow queries
level:warn AND msg:"Slow database query"

# Find failed jobs
level:error AND component:worker

# Find webhook failures
level:error AND webhook:shopify
```

### Alerts

Set up alerts for:

1. **High Error Rate**
   - Threshold: > 10 errors/minute
   - Action: Page on-call engineer

2. **Database Connection Failures**
   - Threshold: Any failure
   - Action: Immediate notification

3. **Job Queue Backlog**
   - Threshold: > 1000 pending jobs
   - Action: Scale worker instances

4. **Shopify API Rate Limiting**
   - Threshold: 80% of rate limit
   - Action: Enable throttling

## Scaling Considerations

### Horizontal Scaling

**Web Service:**
```bash
# Render
- Scale to 2+ instances in Dashboard
- Load balanced automatically

# Railway
- Add replicas in Settings
- Uses round-robin load balancing

# Fly.io
flyctl scale count 3 --region sjc,ord,fra
```

**Background Workers:**
```bash
# Scale workers to handle job volume
# Rule of thumb: 1 worker per 100 jobs/minute
```

### Vertical Scaling

**When to upgrade:**
- Web: Memory > 80% consistently
- Worker: Job processing time increasing
- DB: Connection pool exhaustion
- Redis: Memory > 80%

**Recommended production specs:**
- Web: 2GB RAM, 1 CPU
- Worker: 1GB RAM, 1 CPU per worker
- PostgreSQL: 4GB RAM, 2 CPU
- Redis: 1GB memory

### Database Optimization

**Connection Pooling:**
```

?connection_limit=10&pool_timeout=20
```

## Troubleshooting

### Common Issues

#### 1. Database Connection Errors

**Symptom:**
```
Error: P1001: Can't reach database server
```

**Solutions:**
```bash
# Check DATABASE_URL format
echo $DATABASE_URL

# Verify network access (IP allowlist)
# Render: Allow all Render IPs
# Railway: Auto-configured
# Fly.io: Use internal DNS

# Test connection
pnpm prisma db pull
```

#### 2. Redis Connection Errors

**Symptom:**
```
Error: connect ECONNREFUSED
```

**Solutions:**
```bash
# Check REDIS_URL
echo $REDIS_URL

# Verify Redis is running
redis-cli -u $REDIS_URL ping

# Check SSL requirement
# Some providers require rediss:// (SSL)
```

#### 3. Webhook Verification Failures

**Symptom:**
```
Webhook signature verification failed
```

**Solutions:**
```bash
# Verify SHOPIFY_WEBHOOK_SECRET matches Shopify settings
# Check Shopify Admin → Settings → Notifications → Webhooks

# Ensure raw body is being parsed
# Already handled in middleware.ts
```

#### 4. Build Failures

**Symptom:**
```
Error: Cannot find module '@prisma/client'
```

**Solutions:**
```bash
# Ensure build command includes prisma:generate
# render.yaml already includes this
# Manual fix:
pnpm prisma generate && pnpm build
```

#### 5. Worker Not Processing Jobs

**Symptom:**
Jobs stuck in queue, not being processed.

**Solutions:**
```bash
# Check worker logs
# Should see: "Worker started successfully"

# Verify Redis connection
# Check worker has same REDIS_URL as web

# Restart worker service
# Platform dashboard → Restart
```

#### 6. Shopify OAuth Errors

**Symptom:**
```
Error: redirect_uri mismatch
```

**Solutions:**
```bash
# Verify NEXTAUTH_URL matches Shopify app settings
# Shopify Partners → Apps → Your App → App Setup

# Redirect URL should be:
# https://your-app.com/api/auth/callback/shopify

# Update both:
# - Environment variable: NEXTAUTH_URL
# - Shopify app settings
```
